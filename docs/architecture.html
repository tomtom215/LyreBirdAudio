Design Philosophy

Why Bash?

Bash is pre-installed on virtually every Linux system. Choosing Bash means: no pip/npm/gem conflicts, no version mismatches, no build processes, no containers. Deployment is: git clone, chmod +x, run.

This choice eliminates entire categories of deployment failures that are common with Python, Node.js, or Ruby-based tools — especially on minimal embedded systems like Raspberry Pi.

Why Not Docker?

USB device management requires host-level udev rule creation and direct USB subsystem access. Containers add complexity that conflicts with physical USB port mapping, which is a core feature. LyreBirdAudio is designed for dedicated bare-metal hosts where systemd services provide sufficient process isolation.

Single-Responsibility Principle

Each script has exactly one domain:

  • Orchestrator: User interface and delegation only — no business logic
  • Stream Manager: FFmpeg process management
  • USB Mapper: Device persistence via udev
  • Capability Checker: Hardware capability detection
  • Updater: Version management
  • Diagnostics: System health validation
  • Installer: MediaMTX lifecycle

This modularity prevents duplicate business logic, simplifies testing, and ensures maintainability.

Data Flow

Stream Startup Sequence

  1. Stream manager reads /etc/mediamtx/audio-devices.conf
  2. Scans /dev/snd/by-usb-port/ for available devices
  3. For each device, looks up config (friendly name → full ID → defaults)
  4. Starts MediaMTX if not running
  5. Launches FFmpeg wrapper for each device
  6. Wrapper supervises FFmpeg with exponential backoff restart logic
  7. MediaMTX API confirms stream is active
  8. Cron job monitors health every 5 minutes via monitor command

Exponential Backoff Recovery

Attempt 1: Wait 10s before restart
Attempt 2: Wait 20s
Attempt 3: Wait 40s
...
Max delay: 300s (5 minutes)

After 300s of stable operation: reset backoff counter
After MAX_CONSECUTIVE_FAILURES: alert and stop retrying

USB Device Persistence Flow

  1. USB device connected to physical port
  2. Kernel assigns /dev/snd/pcmC0D0c (order may vary)
  3. udev rule matches: SUBSYSTEM=="sound", ATTRS{idVendor}=="XXXX", ENV{ID_PATH}=="..."
  4. udev creates symlink: /dev/snd/by-usb-port/Device_1 → ../pcmC0D0c
  5. FFmpeg uses symlink path — always /dev/snd/by-usb-port/Device_1
  6. On next boot: symlink is recreated in same location regardless of enumeration order

Architecture Decision Records

Key technical decisions and their rationale:

ADR-001: Bash as Primary Language

Decision
Bash 4.0+ for all scripts
Rationale
Universal availability on Linux, zero runtime dependencies, native system integration, easy to audit, familiar to sysadmins
Trade-offs
Limited to Linux/Unix; more verbose than Python for complex logic; requires careful quoting discipline

ADR-002: MediaMTX as RTSP Server

Decision
MediaMTX (formerly rtsp-simple-server) as the RTSP server
Rationale
Single binary deployment, low resource usage (suitable for Pi), REST API for programmatic control, supports RTSP/WebRTC/HLS, active development
Trade-offs
External dependency; must handle API version changes

ADR-003: FFmpeg for Audio Transcoding

Decision
FFmpeg as the audio capture and encoding pipeline
Rationale
Native ALSA support, codec flexibility, battle-tested stability, configurable for quality/performance trade-offs
Trade-offs
Must be installed; process lifecycle management complexity

ADR-004: Shared Library Pattern

Decision
lyrebird-common.sh sourced by all scripts
Rationale
DRY principle, consistent logging format, backward compatibility via override pattern
Trade-offs
Must maintain backward compatibility; source order matters

ADR-005: Atomic File Operations

Decision
Write to .tmp file, then mv to final location
Rationale
Prevents partial reads by concurrent processes; POSIX guarantees mv on same filesystem is atomic; crash-safe
Trade-offs
Slightly more complex code; stale .tmp cleanup needed

ADR-006: udev for Device Persistence

Decision
udev rules create persistent symlinks by USB port topology
Rationale
Kernel-integrated, no daemon required, survives reboots and reconnections
Trade-offs
Root required to install rules; USB hub changes require remapping; reboot needed

ADR-007: Webhook-Based Alerting

Decision
Pure-Bash webhook support for Discord, Slack, ntfy, Pushover
Rationale
Users choose their own notification platform; no email/SMS infrastructure required; easy to extend
Trade-offs
Requires network connectivity; rate limiting needed to prevent spam

ADR-008: Prometheus Metrics Format

Decision
Export metrics in Prometheus/OpenMetrics text format
Rationale
Industry standard, plain text (easy to debug), wide ecosystem (Grafana, AlertManager)
Trade-offs
Must maintain metric naming conventions; metric cardinality control required

ADR-009: Lockfile Concurrency Control

Decision
Lockfiles prevent concurrent script execution
Rationale
Simple, debuggable, atomic acquisition via mkdir or flock
Trade-offs
Stale locks possible after crash; cleanup logic required

ADR-010: Signal Handler Pattern

Decision
trap cleanup EXIT INT TERM HUP QUIT in all long-running scripts
Rationale
Ensures cleanup (remove locks, kill child processes) on all exit paths
Trade-offs
More complex code; cleanup must be idempotent
