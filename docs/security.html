Security Model Overview

LyreBirdAudio has three main network interfaces:

InterfaceDefault PortPurposeDefault Security
RTSP 8554 Audio stream consumption Open (no auth)
MediaMTX API 9997 Stream management REST API Localhost only
WebRTC 8889 Browser-based streaming Open (no auth)

Default Security Posture

Out of the box, LyreBirdAudio implements:

  • No hardcoded credentials — all authentication delegated to MediaMTX
  • Restrictive permissions — config files mode 640, state directories 750
  • Service user isolation — MediaMTX runs as dedicated mediamtx user
  • systemd hardening — NoNewPrivileges, PrivateTmp, ProtectSystem
  • Input validation — all user inputs sanitized against injection
  • Safe temp files — mktemp used for all temporary files
  • Download verification — SHA256 checksums verify MediaMTX binaries
  • API localhost-only — MediaMTX REST API bound to 127.0.0.1 by default

Security Checklist

Minimal Security (Home Use)

  • ☐ Keep system updated: sudo apt update && sudo apt upgrade
  • ☐ Use firewall to block external access if streams are local-only
  • ☐ Don't expose ports to the internet unnecessarily

Standard Security (Remote Access)

  • ☐ All minimal items above
  • ☐ Enable RTSP authentication
  • ☐ Restrict API to localhost or specific IPs
  • ☐ Configure firewall rules to limit access
  • ☐ Enable log rotation

Enhanced Security (Field Deployment)

  • ☐ All standard items above
  • ☐ Enable TLS encryption (RTSPS)
  • ☐ Use VPN for remote access instead of direct exposure
  • ☐ Enable audit logging
  • ☐ Implement physical security measures
  • ☐ Schedule regular security update runs

RTSP Authentication

Internal Authentication (Recommended)

# Edit /etc/mediamtx/mediamtx.yml

authMethod: internal

authInternalUsers:
  # Admin user - can publish and read all streams
  - user: admin
    pass: your_secure_password_here
    permissions:
      - action: publish
      - action: read
      - action: playback
      - action: api

  # Read-only viewer
  - user: viewer
    pass: viewer_password_here
    permissions:
      - action: read

JWT Authentication

# Edit /etc/mediamtx/mediamtx.yml

authMethod: jwt
authJWTJWKS: https://your-auth-server/.well-known/jwks.json

Client Usage with Authentication

# VLC / FFplay with credentials
ffplay rtsp://username:password@your-server:8554/Device_1
vlc rtsp://username:password@your-server:8554/Device_1

# FFmpeg recording with auth
ffmpeg -i rtsp://viewer:pass@server:8554/Device_1 -c copy out.mkv

Network Security

UFW Firewall Rules

# Allow RTSP only from your local network
sudo ufw allow from 192.168.1.0/24 to any port 8554 proto tcp

# Block external RTSP access entirely
sudo ufw deny 8554

# Keep API localhost-only (deny external)
sudo ufw deny 9997

# Enable UFW
sudo ufw enable
sudo ufw status

Bind MediaMTX to Specific Interface

# /etc/mediamtx/mediamtx.yml

# Only listen on specific interface (not all)
rtspAddress: 192.168.1.100:8554

# Keep API on localhost only (already default)
apiAddress: 127.0.0.1:9997

VPN Access (Recommended for Remote Monitoring)

Instead of exposing RTSP publicly, connect remote viewers through a VPN. This is more secure and avoids the complexity of TLS certificate management. 

# Install WireGuard (recommended VPN)
sudo apt install wireguard

# All streams accessible within VPN network
# rtsp://10.0.0.1:8554/Device_1  (VPN IP)
# No need to expose ports to public internet

API Security

The MediaMTX REST API should remain localhost-only unless you have specific integration needs:

# /etc/mediamtx/mediamtx.yml

# Localhost only (default, recommended)
apiAddress: 127.0.0.1:9997

# If you need LAN access, restrict to specific IP:
# apiAddress: 192.168.1.100:9997

File System Security

# Verify configuration file permissions
stat /etc/mediamtx/audio-devices.conf
# Should be: -rw-r----- (640) owned by root:mediamtx

# Fix permissions if needed
sudo chmod 640 /etc/mediamtx/audio-devices.conf
sudo chown root:mediamtx /etc/mediamtx/audio-devices.conf

# Secure the config directory
sudo chmod 750 /etc/mediamtx
sudo chown root:mediamtx /etc/mediamtx

Library Integrity Verification

For high-security deployments, verify the shared library before execution:

# Generate hash of lyrebird-common.sh
HASH=$(sha256sum lyrebird-common.sh | cut -d' ' -f1)
echo "Expected hash: $HASH"

# Set in environment or systemd service override
export LYREBIRD_COMMON_EXPECTED_HASH="$HASH"

# Scripts will verify the library before sourcing it
sudo ./lyrebird-stream-manager.sh start

Field Deployment Recommendations

ℹ️
Wildlife Monitoring Context

These recommendations are particularly relevant for unattended field deployments — remote wildlife monitoring stations, environmental sensors, or any system with limited physical access and potential exposure to untrusted networks.

  1. Use VPN — Connect remote units via WireGuard or OpenVPN rather than exposing RTSP directly to the internet.
  2. Enable authentication — Add internal users with strong passwords to MediaMTX if VPN isn't available.
  3. Regular updates — Schedule monthly: ./lyrebird-updater.sh and sudo apt upgrade.
  4. Review logs — Check /var/log/lyrebird/ monthly for anomalies.
  5. Physical security — Secure the device against physical access. USB port tamper detection is not built-in.
  6. Emergency wipe plan — For sensitive deployments, keep a plan for remote wiping if a unit is compromised.
  7. Disable unused ports — Block WebRTC (8889) if not needed: sudo ufw deny 8889.
  8. Log rotation — Ensure logrotate is configured to prevent disk exhaustion: sudo cp config/lyrebird-logrotate.conf /etc/logrotate.d/lyrebird.

For comprehensive security guidance, see the full SECURITY-GUIDE.md in the repository.
